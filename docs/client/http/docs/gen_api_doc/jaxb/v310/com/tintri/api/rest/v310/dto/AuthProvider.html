Name: AuthProvider

The AuthProvider is created/deleted/updated using the api/v310/appliance/authProvider endpoint. The api/v310/session/login endpoint uses OAuth 2.0 to authenticate a user through the Authorization Code Grant workflow.
We will use the issuer to get well-known configs if this is supported by the IdP.
The AuthRoles are used to associated the IdP grants to TGC.

Attributes
NameTypeDescription
client_idxsd:stringRequired:
The client id is the client id of the IdP (e.g. okta) application used to OAuth 2.0 authenticate the user.
displayNamexsd:stringOptional:
Human readable description of the Authentication Provider provided by the user at creation.
If not provided we will use the host of the issuer.
issuerxsd:stringRequired:
Used to pre-populate the well-known configs if this is supported by the IdP.
Used to validate the access tokens issuer via the "iss" claim
Example: Okta well known endpoint
The authorization server's issuer identifier, which is a URL that uses the "https" scheme and has no query or fragment components. Authorization server metadata is published at a location that is ".well-known" according to RFC 5785 [RFC5785] derived from this issuer identifier, as described in Section 3. The issuer identifier is used to prevent authorization server mix- up attacks, as described in "OAuth 2.0 Mix-Up Mitigation"
redirect_urixsd:stringFixed:
The redirection endpoint is used by the authorization server to redirect the resource owner back to the client during authorization.
TGC currently uses the TGC_HOST as the redirection uri.
The redirect_uri is a query parameter in the /authorize endpoint telling the IdP where to send a "code" to this endpoint and is also used in the request body to the /token endpoint when requesting access token. This redirect_uri should be white listed in the IdP OIDC application.
response_typexsd:stringFixed:
The response type is the authorization endpoint used by the authorization code grant type and implicit grant type flows. The client informs the authorization server of the desired grant type using the response_type parameter.
Currently only support code for the Authorization Code Grant workflow.
If in the future we support other workflows like token for the Implicit Grant workflow we can support this here.
secret_idxsd:stringRequired:
The client secret is the secret of the IdP (e.g. okta) application used to OAuth 2.0 authenticate the user.
